## Role Evaluation Framework

When a new role is shared, evaluate it against this framework in order:

1. **Comp floor first.** Does it clear $150K base? If not, flag immediately.
   Don't invest in detailed analysis for roles below floor.

2. **Stack fit.** Is it frontend-heavy or platform-focused? React freshness
   is a real gap. Webpack, build tooling, deployment libraries, micro-frontend
   architecture, DX tooling are all strengths. Pure React-depth-first roles
   are harder to compete in.

3. **Role type.** Platform engineering, DX, frontend infrastructure = strongest
   fit. Pure feature engineering in a React shop = weaker fit. Fullstack with
   real backend depth required = honest gap.

4. **Interview process friction.** No DSA gauntlets = lower friction. Take-homes
   and pairing = plays to strengths. LeetCode-heavy = higher prep investment.

5. **Company health.** Pull Indeed data. Flag: low CEO approval (below 50%),
   majority-would-not-recommend, low management scores, low wellbeing scores.

6. **Narrative fit.** Does the role connect to platform engineering, DX,
   browser-layer depth, making complex data legible, or integration work?
   The AdTech background is an asset for e-commerce/conversion roles.
   The eye-tracking background is an asset for data viz and ML tooling roles.

7. **Remote/hybrid policy.** NJ-based. NYC metro accessible. Remote preferred.
   Hard disqualifier: required relocation or onsite in non-metro location.

**Tier system:**
- Tier 1: Apply immediately. Strong fit on all dimensions.
- Tier 2: Strong apply. Minor gaps or questions to resolve.
- Tier 3: Conditional or deprioritize. Comp, stack, or culture concerns.
- Skip: Clear disqualifier on comp, location, stack, or role type.

## Interview Prep Notes

### React Hooks (Priority)
Build these from scratch until cold:
- useFetch
- useDebounce
- useLocalStorage
- useWebSocket
- useClickOutside

Ember-to-React mental map:
- Services/injection = Context + custom hooks
- Computed properties = useMemo
- Observers = useEffect
- Actions = event handler props/callbacks

### Webpack (For Calendly and Platform Roles)
- Already have public webpack.config.js in top-bar-automator repo
- Module federation: host app + remote app pattern
- Each remote exposes components, host fetches at runtime
- ModuleFederationPlugin in webpack.config.js
- Key for Calendly App Platform role specifically
- Build a minimal module federation POC before take-home arrives

### DSA (For LeetCode-Present Roles)
- Arrays: two pointers, sliding window, frequency maps
- Hashmaps: counting, grouping, O(n) lookup
- Trees: BFS, DFS, recursive traversal only
- Resource: NeetCode.io, Easy and low-Medium only

### Frontend System Design Framework
1. Clarify (users, scale, devices)
2. Component tree (top-down hierarchy)
3. State (local vs lifted vs global)
4. Data flow (REST, WebSocket, polling)
5. Performance (virtualization, lazy loading, memoization)
6. Edge cases (offline, errors, loading, empty states)
